逆・裏・対偶 論理ハッキングRPG 要件定義書

1. プロジェクト概要

ジャンル: 2D探索＆コマンドバトルRPG（論理パズル要素融合型）

コア体験: 「逆・裏・対偶」の論理演算を用いて、ゲーム内のオブジェクトの属性（タグ）やイベント進行を規定する『世界のルール（命題）』をハッキングし、正規ルートを論理的に逸脱していくメタフィクション体験。

プラットフォーム/エンジン: Unity（2D）

ターゲット層: 論理パズル（『Baba Is You』等）のシステム解法や、メタ展開によるシナリオの裏かきを好むプレイヤー。

トーン＆マナー: SFC時代の王道2Dドット絵RPGをベースとする。ただし、ルール改変UI操作時や事象の強制書き換え発生時には、無機質なシステムフォントやグリッチ演出を用い、意図的な「バグ（ハッキング）」の違和感を演出する。

想定プレイ時間: 約30分。RPGとしての成長要素は最小限に留める。アドホックなイベント処理（専用スクリプト）を排し、全てを普遍的なルール評価エンジンの挙動として構築することで、プレイヤーの想定外の論理的解法も「正答」として許容する密度の高い体験とする。

2. ゲームシステム要件

2.1. シームレスなルール改変（高自由度設計）

ルールの段階的解放（ページ概念）: ルールブックは初めから全てを書き換えられるわけではなく、進行に応じて「ページ」が追加される。

第1章「状態異常の法則」（例：毒、炎のタグ付与に関する命題）

第2章「物理と空間の法則」（例：扉の開閉状態、地形タグの書き換え条件）

最終章「世界の真理」（例：重要NPCの生存状態、勝利条件の定義）

スコープの広域化と評価タイミング: プレイヤーがルールを書き換えた状態は、フィールド探索中・バトル中問わず維持される。ルールの再評価（Rule Evaluatorによる実行）は、「プレイヤーが1アクション（移動・コマンド実行）を終えた直後のフレーム」で一斉に行われる。

詰み（ソフトロック）防止策: ペナルティなしの「システムの再起動（ルール初期化）」を常時可能とする。

フレーバー: UI上は「デバッグモード：設定の初期化」といったメタ的な表現を用い、プレイヤーが世界を壊しすぎた場合のセーフティネットとする。

2.2. フィールド・バトル統合システム（グリッド＆同期型ターン制）

探索とバトルをシームレスに繋ぎ、ルール評価を厳密に行うため、マス目上のマップと同期型のターン進行を採用する。（※パーマデスやランダム生成等のローグライク要素は含まない）

時間の統一: プレイヤーが「1マス移動する」または「1つのコマンド（たたかう、アイテム）を実行する」ことで1ターンが消費され、同時に世界全体（敵の移動・攻撃、ギミックの作動状態、継続タグの処理）も1ターン分進行する。

アクションのタグ化（コマンドの再定義）:

「たたかう」：対象のマスに [Damage: Physical] タグを1ターンの間発生させる空間干渉行為。

「アイテム」：火炎瓶なら対象マスに [Element: Fire] [Duration: 3Turns] タグを付与する行為。

ギミック解決のフロー例

論理的飛躍（専用イベント等）を排除し、評価エンジンによるタグ（状態）の強制操作によって事象をコントロールする。

例1：事象の強制生成（逆の成功例1）

初期状態: 「対象マスに [火属性魔法] が発生する」ならば、「 [氷スライム] は [即死] する」

改変（逆）: 「 [氷スライム] が [即死] する」ならば、「対象マスに [火属性魔法] が発生する」

アクション: プレイヤーが強力な物理攻撃アイテム等を使い、氷スライムを一撃で倒す（即死条件を満たす）。

結果: スライムが死んだ事実をシステムが検知し、論理的整合性を保つために**「スライムが死んだマス」に無から [火属性魔法（炎のオブジェクト等）] を強制発生させる**。主人公が魔法を覚えるわけではなく、環境に炎が残るため、これを連鎖させて隣接する氷の壁を溶かすパズルに利用する。

例2：空間のハッキング（逆の成功例2）

初期状態: 「対象が [毒沼マス] 上にいる」ならば、「対象は [継続ダメージ] を受ける」

改変（逆）: 「対象が [継続ダメージ] を受ける」ならば、「対象は [毒沼マス] 上にいる」

アクション: プレイヤーが自らに「毒薬」や「火炎瓶」などのアイテムを使用し、意図的に [継続ダメージ] （状態異常タグ）を付与する。

結果: システムが「対象が継続ダメージを受けている」事実を検知し、整合性を保つためにプレイヤーの足元のマスを強制的に [毒沼マス] に書き換える（あるいは最寄りの毒沼マスへ強制ワープさせる）。本来進入不可能な地形（水上や壁など）を毒沼に変換して道を作るなど、空間操作のパズルとして機能する。

例3：確率の強制固定化（裏の成功例）

初期状態: 「ボスのHPが1になる」ならば、「ボスは [即死攻撃] を打ってくる」
(※論理学上、この命題は「HPが2以上の時に即死攻撃を打ってこない」ことを保証しない。実際、このボスはHPに関わらずランダムで即死攻撃を連発してくる理不尽な設定とする)

改変（裏）: 「ボスのHPが1にならない」ならば、「ボスは [即死攻撃] を打ってこない」

アクション: プレイヤーが「固定2ダメージ」を与えるアイテムのみを使用して攻撃し、ボスのHPを奇数から削り続け、HP1の状態をスキップして倒す。

結果: 「HPが1にならない」という条件が常に真となるため、システムの強制力によりボスの即死攻撃の発動確率が0%に固定される。ランダム行動を封殺し、安全に撃破できる。

例4：メタ的なシナリオ分岐（裏のシナリオハッキング例）

初期状態: 「勇者が勝つ」ならば、「魔王は死ぬ」

改変（裏）: 「勇者が勝たない」ならば、「魔王は死なない」

アクション: 戦闘中、勇者は魔王のHPを1まで削り、会話イベントを発生させる。魔王は改心し真の悪（システム）について語り始める。その瞬間、システム側が強制的に魔王に [即死] タグを付与しようとする（原因不明の死）が、プレイヤーはトドメを刺さず「防御」や「回復」を選択して「勝たない」状態を維持する。

結果: Rule Evaluatorが「勇者が勝たない」状態を検知し、システムの強制力として**「魔王は死なない（[即死]タグの付与を拒絶する）」状態を絶対的に保護**する。システムによる暗殺処理はルール干渉によってブロックされ、魔王が生存したまま隠しルートへ分岐する。

3. UI/UX要件（ルール改変インターフェース）

3.1. 命題のモジュール化

文章を視覚的なブロックとして分割し、キャンバス上に配置する。

条件ブロック（P）: [ ≡ (持ち手) ] 主人公が魔王に聖剣を突き刺す [ 🔄 (反転) ]

接続ブロック: [ と、 / ならば、 ] （固定領域・操作不可）

帰結ブロック（Q）: [ ≡ (持ち手) ] 魔王は死ぬ [ 🔄 (反転) ]

3.2. インタラクション仕様

位置の入れ替え（逆）: 各ブロックの「持ち手」をドラッグ＆ドロップし、条件と帰結の位置をスワップする。

真偽の反転（裏）: 各ブロックの「反転ボタン」をクリックし、テキストを否定形に書き換え、視覚的エフェクト（暗転、取り消し線など）を適用する。

複合操作（対偶）: 「入れ替え」と「両方の反転」を組み合わせることで対偶を成立させる。

4. シナリオ・世界観要件

序盤（偽装段階）: ルールブックは「絶対的な攻略手順」として機能し、プレイヤーに王道RPGの法則を刷り込む。

中盤（覚醒段階）: 主人公がルールの書き換え（論理演算）能力を獲得する。以降、ルールブックは「世界の法則（タグの評価基準）を書き換えるツール」へ変化する。

終盤（メタ展開）: シナリオ進行を規定する命題（例：「魔王は死ぬ」）の真偽をシステムレベルで改変・保護することで、真の黒幕の存在や隠しルートへシナリオを分岐させる。

5. システムアーキテクチャ要件（システミック・デザイン）

高度な自由度と安定性を両立するため、イベント駆動型（If-Else）のスクリプトによる個別処理を極力排し、以下の基盤を採用する。

5.1. タグベースの属性管理（Entity Component System指向）

キャラクター、アイテム、フィールド上のオブジェクトはすべて「タグ（Tag）」と「状態（State）」の集合体として定義する。

例：火炎瓶＝[属性:炎][効果:継続ダメージ]、魔王＝[種族:ボス][状態:可燃性]

5.2. グローバル・ルール評価エンジン（Rule Evaluator）

ゲーム全体を監視する独立した評価エンジンを実装する。

プレイヤーがルールブックを書き換えた瞬間、または世界の状態（変数の値、タグの付与状況）が変化したターンの終わりに、現在の「命題（PならばQ）」が成立しているかを全オブジェクトに対して一斉評価する。

評価が真（True）となった場合、対象となるオブジェクトに対して機械的に「結果（Q）」を強制適用（タグの付与・削除、座標の書き換え等）する。

5.3. 実装要件（UI制御）

ドラッグ操作（IDragHandler）とクリック操作（Button）の処理領域をGameObjectレベルで物理的に分離し、競合を回避する。

各命題パネルは「現在位置（条件部/帰結部）」および「否定状態フラグ」を保持し、Rule Evaluatorへ現在の論理状態を送信する。
